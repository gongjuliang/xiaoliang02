# NATT 内网穿透使用文档

本文说明如何启动、初始化、配置和使用 `nattserver` 服务端与 `nattuser` 客户端。当前版本只支持 TCP 端口穿透，通信模型为“控制连接 + 数据连接分离”。

## 1. 项目目录

```text
D:\2025Code\ai\natt
  nattserver\    服务端
  nattuser\      客户端
  commonGin\     参考项目
```

服务端默认文件：

```text
nattserver\config\config.json
nattserver\data\nattserver.db
nattserver\logs\YYYY-MM-DD.log
nattserver\logs\audit\YYYY-MM-DD.jsonl
```

客户端默认文件：

```text
nattuser\config\config.json
nattuser\data\nattuser.db
nattuser\logs\YYYY-MM-DD.log
nattuser\logs\audit\YYYY-MM-DD.jsonl
```

## 2. 默认端口

服务端：

| 用途 | 默认值 |
| --- | --- |
| Web 管理后台 | `0.0.0.0:25510` |
| MCP 接口 | `0.0.0.0:25510/mcp` |
| 客户端控制连接 | `0.0.0.0:25511` |
| 客户端数据连接 | `0.0.0.0:25512` |
| 公网映射端口范围 | `0-65535` |

客户端：

| 用途 | 默认值 |
| --- | --- |
| 本地 Web 管理后台 | `127.0.0.1:25520` |
| MCP 接口 | `127.0.0.1:25520/mcp` |

启动前会检查本机需要监听的端口是否被占用。如果端口被占用，会输出类似 `25510端口被占用` 的中文错误。

## 3. 首次初始化

默认启动时，如果 `config/config.json` 不存在，或数据库不存在、`users` 表为空，程序会进入初始化页面。

服务端初始化页：

```text
http://127.0.0.1:25510/init.html
```

客户端初始化页：

```text
http://127.0.0.1:25520/init.html
```

初始化页面需要填写：

- 控制台管理员账号，不能为空。
- 控制台管理员密码，至少 8 位，必须同时包含字母和数字。
- 运行模式：测试模式对应 `app.environment=development`，生产模式对应 `app.environment=production`。
- 勾选“已阅读并同意《用户协议》”。

初始化成功后会自动创建：

- `config/config.json`
- `data/`
- `logs/`
- SM2 密钥文件
- SQLite 数据库和管理员账号

项目不再内置默认账号 `admin` 和默认密码 `admin123456`。

## 4. 启动方式

启动服务端：

```powershell
cd D:\2025Code\ai\natt\nattserver
go run . -config config/config.json
```

访问服务端后台：

```text
http://127.0.0.1:25510/login.html
```

启动客户端：

```powershell
cd D:\2025Code\ai\natt\nattuser
go run . -config config/config.json
```

访问客户端后台：

```text
http://127.0.0.1:25520/login.html
```

显式传入 YAML 配置仍兼容：

```powershell
go run . -config config/config.yaml
```

不传 `-config` 时默认读取 `config/config.json`。默认启动时缺少 JSON 配置会进入初始化页面；显式传入 `-config` 时不会进入交互初始化。

## 5. 登录与安全

Web 登录使用验证码和 SM2 前端加密：

- 登录页会先请求 `/api/*/v1/auth/sm2-public-key` 获取 SM2 公钥。
- 前端使用内置离线 `sm2.js` 加密密码后再提交。
- 生产模式不接受明文密码。
- 登录验证码通过图片接口展示，页面 DOM 不包含算式明文。
- 同一 IP 登录失败每累计 10 次会触发递进封禁：5 分钟、10 分钟、30 分钟、1 小时、6 小时；重启程序后封禁记录清空。

JWT 登录态用于后台接口鉴权。密码和秘钥哈希使用当前项目的 SM3 哈希格式。

## 6. 基本穿透流程

### 6.1 服务端创建客户端授权

1. 打开服务端后台：`http://127.0.0.1:25510/login.html`。
2. 登录后进入“客户端”页面。
3. 新增客户端授权。
4. 保存后记录弹出的客户端秘钥。

### 6.2 客户端配置服务端连接

1. 打开客户端后台：`http://127.0.0.1:25520/login.html`。
2. 进入“隧道连接”页面。
3. 新增服务端连接，填写服务端地址、控制端口 `25511`、数据端口 `25512`、服务端秘钥、本地目标地址和本地目标端口。
4. 保存后点击连接。

如果秘钥错误、隧道被占用、服务端暂停隧道等显性错误，会以中文写入状态详情。

### 6.3 服务端创建公网隧道

服务端隧道只管理公网监听端口和状态，不再填写内网本地地址和本地端口。

常用字段：

| 字段 | 示例 | 说明 |
| --- | --- | --- |
| 名称 | `ssh-home` | 隧道名称 |
| 公网监听地址 | `0.0.0.0` | 服务端监听地址 |
| 公网监听端口 | `10022` | 外部访问端口，必须在 `1-65535` 之间且未被占用 |
| 自启动 | 按需 | 客户端重连后是否自动恢复 |
| 状态详情 | 中文错误或 `-` | 展示最近一次错误 |

服务端创建隧道后，客户端需要使用对应的服务端隧道 ID 配置本地目标。

### 6.4 客户端绑定本地目标

客户端隧道连接配置中保存本地目标：

| 字段 | 示例 | 说明 |
| --- | --- | --- |
| 服务端隧道 ID | `1` | 服务端创建的隧道 ID |
| 本地地址 | `127.0.0.1` | 客户端机器可访问的内网服务地址 |
| 本地端口 | `22` | 客户端机器上的内网服务端口 |
| 服务端秘钥 | `xiaoliang_...` | 前端默认遮挡，可点击查看 |

绑定完成并连接成功后，外部访问：

```text
服务端公网 IP:公网监听端口
```

会被转发到：

```text
客户端本地地址:客户端本地端口
```

示例：

```text
1.2.3.4:10022 -> 127.0.0.1:22
```

## 7. 配置说明

默认配置格式是 JSON：

```text
config/config.json
```

JSON 标准不支持注释，配置说明请写在部署文档里。旧 YAML 配置文件仍可通过显式 `-config config/config.yaml` 读取。

配置页面只允许编辑可热更新配置。不可热更新配置只读展示，如需修改需要编辑配置文件后重启程序。

TLS 相关配置已经从当前业务入口删除；客户端和服务端通信使用普通 TCP。

## 8. MCP 调用

MCP 使用标准 Streamable HTTP JSON-RPC，服务端和客户端都挂在各自 HTTP 端口的 `/mcp` 路径下，不再使用旧的 `/mcp/tools/call`。

服务端 MCP：

```text
POST http://127.0.0.1:25510/mcp
Authorization: Bearer replace-with-server-mcp-token
Content-Type: application/json
```

客户端 MCP：

```text
POST http://127.0.0.1:25520/mcp
Authorization: Bearer replace-with-client-mcp-token
Content-Type: application/json
```

初始化示例：

```json
{
  "jsonrpc": "2.0",
  "id": 1,
  "method": "initialize",
  "params": {
    "protocolVersion": "2025-11-25",
    "capabilities": {},
    "clientInfo": {
      "name": "codex",
      "version": "1.0.0"
    }
  }
}
```

工具发现示例：

```json
{
  "jsonrpc": "2.0",
  "id": 2,
  "method": "tools/list",
  "params": {}
}
```

工具调用示例：

```json
{
  "jsonrpc": "2.0",
  "id": 3,
  "method": "tools/call",
  "params": {
    "name": "server.list_clients",
    "arguments": {
      "page": 1,
      "page_size": 20
    }
  }
}
```

成功返回示例：

```json
{
  "jsonrpc": "2.0",
  "id": 3,
  "result": {
    "content": [
      {
        "type": "text",
        "text": "{...}"
      }
    ],
    "structuredContent": {},
    "isError": false
  }
}
```

工具业务失败会返回 `result.isError=true`；JSON-RPC 请求格式错误、未知方法等协议错误会返回 `error.code` 和 `error.message`。

### 8.1 自动生成测试证书时的 HTTPS 注意事项

如果初始化时选择了“自动生成测试证书”，调用 `https://127.0.0.1:25510/mcp` 可能会看到类似下面的错误：

```text
http: TLS handshake error from 127.0.0.1:63201: remote error: tls: unknown certificate
```

这不是 MCP 接口本身坏了，而是因为自动生成的测试证书属于自签名证书，调用方默认不信任它。服务端日志里的 `TLS handshake error` 只是表示客户端在握手阶段拒绝了这个证书。

如果你想继续使用 HTTPS，可以按下面的方式处理：

1. 打开 `ssl\web.crt`
2. 双击证书文件，选择“安装证书”
3. 证书存储选择“将所有证书放入下列存储”
4. 选择“受信任的根证书颁发机构”
5. 完成安装后，重启浏览器或 Codex 客户端
6. 重新访问 `POST https://127.0.0.1:25510/mcp`

注意：

- 标准 MCP 调用路径是 `POST /mcp`
- `https://127.0.0.1:25510/mcp/` 也许能被某些客户端容忍，但建议统一使用不带尾部斜杠的 `/mcp`
- 如果你只是本机调试，也可以直接使用 `http://127.0.0.1:25510/mcp`

## 9. 审计与日志

普通运行日志：

```text
nattserver\logs\YYYY-MM-DD.log
nattuser\logs\YYYY-MM-DD.log
```

普通日志包含触发位置，例如：

```text
[INFO] routes.go:61 server routes registered
```

审计日志写入 JSONL 文件：

```text
nattserver\logs\audit\YYYY-MM-DD.jsonl
nattuser\logs\audit\YYYY-MM-DD.jsonl
```

Web 审计页面和 `/audit-logs` API 读取 JSONL 文件分页展示。MCP 工具发现和工具调用也会写入审计日志，并对 token、secret、password、key 等敏感参数做脱敏。

## 12. 常见问题

隧道启动失败：

- 确认客户端在线。
- 确认公网监听端口在允许范围内且未被占用。
- 确认客户端已为对应服务端隧道 ID 配置本地目标。
- 确认本地目标启用，且客户端机器可访问该本地服务。
- 确认服务端防火墙放行公网映射端口。
- 确认客户端可访问服务端数据端口 `25512`。

MCP 返回 unauthorized：

- 确认 `mcp.enabled=true`。
- 确认 MCP Key 已生成。
- 确认请求头为 `Authorization: Bearer <token>` 或 `X-MCP-Token: <token>`。
- 确认调用地址是 HTTP 端口下的 `/mcp`，不是旧的独立 MCP 端口，也不是旧的 `/mcp/tools/call`。

生产模式登录提示“加密密码不正确”：

- 确认浏览器能加载 `/static/js/sm2.js`。
- 确认 `/api/*/v1/auth/sm2-public-key` 能返回 `public_key_hex`。
- 确认没有使用旧缓存页面，必要时强制刷新浏览器缓存。
